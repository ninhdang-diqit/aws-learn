# Hybrid Messaging Architecture Project

Using BOTH RabbitMQ and Kafka Correctly

Goal:
Understand:

* where RabbitMQ is the RIGHT tool
* where Kafka is the RIGHT tool
* why large systems often use both
* difference between task processing and event streaming
* how modern distributed systems separate operational workflows from analytical/event pipelines

---

## PROJECT IDEA

Project name:
Realtime Food Delivery Platform

Inspired by:

* GrabFood
* ShopeeFood
* UberEats

Why this project is perfect:
Food delivery naturally contains:

* task workflows
* retries
* realtime events
* analytics
* tracking
* notifications
* high event volume

Some problems are:
"DO this task"

Others are:
"THIS event happened"

That distinction is exactly RabbitMQ vs Kafka.

## WHY THIS ARCHITECTURE IS REALISTIC

Operational systems need:

* retries
* acknowledgements
* task ownership
* workflow management

=> RabbitMQ is excellent here.

Business intelligence systems need:

* event history
* replay
* streaming
* fanout to many systems
* analytics

=> Kafka is excellent here.

## PHASE 1 — ORDER CREATION

Feature:
Customer places order.

Flow:
Frontend
-> Order Service

Then TWO things happen:

1. RabbitMQ:
   create kitchen processing task

2. Kafka:
   publish OrderCreated event

---

## WHY BOTH?

Kitchen processing:

* specific task
* must be completed once
* retry important
* queue semantics important

=> RabbitMQ

OrderCreated:

* many services interested
* analytics
* audit
* recommendation
* fraud detection
* replay valuable

=> Kafka

## PHASE 2 — RABBITMQ WORKFLOWS

Goal:
Learn operational task processing.

RabbitMQ queues:

* kitchen.jobs
* payment.retry
* push.notifications
* sms.notifications
* email.notifications

Features:

* ACK/NACK
* retry queue
* delayed retry
* DLQ
* priority queue

Experiments:

* worker crash
* retry failures
* poison messages

What to learn:
RabbitMQ is AMAZING for:

* task ownership
* retries
* workflow execution
* delivery guarantees

## PHASE 4 — LIVE ANALYTICS DASHBOARD

Frontend:
Admin dashboard showing:

* orders/minute
* revenue/minute
* active drivers
* failed payments
* top restaurants

Data source:
Kafka stream.

NOT RabbitMQ.

Why?
Because analytics needs:

* continuous stream
* historical replay
* high throughput

---

## PHASE 5 — RETRY COMPARISON

RabbitMQ retry:

* payment retry queue
* exponential backoff
* DLX

Kafka retry:

* retry topics
* dead letter topics

Implement BOTH.

Compare:

* complexity
* readability
* operational simplicity

Critical learning:
RabbitMQ retry is more natural.

---

## PHASE 6 — REPLAY COMPARISON

Kafka:
Add button:
"Replay all events from yesterday"

Analytics rebuilds entirely from Kafka.

RabbitMQ:
Attempt same thing.

Experience:

* difficult
* unnatural
* requires custom storage

Critical learning:
Replay is Kafka's superpower.

---

## PHASE 7 — CONSUMER GROUPS VS WORKERS

RabbitMQ:
Multiple workers consume queue.

Kafka:
Multiple consumers inside consumer group.

Observe:

* partition ownership
* rebalance
* lag

Critical lesson:
Kafka scaling model is partition-based.

## KEY COMPARISON TABLE

USE RABBITMQ FOR:

* email sending
* payment retry
* image processing
* notifications
* workflow execution
* async background jobs
* RPC-style async communication

USE KAFKA FOR:

* analytics
* event history
* realtime streaming
* audit logs
* recommendation systems
* clickstream
* location tracking
* CDC
* monitoring pipelines

---

## MOST IMPORTANT EXPERIMENTS

1. Stop Kafka consumer
   Observe lag buildup.

2. Replay Kafka events
   Rebuild analytics.

3. Crash RabbitMQ worker
   Observe retries + DLQ.

4. Produce massive location events
   See Kafka throughput advantages.

5. Try replaying RabbitMQ tasks
   Feel why it is awkward.

6. Add new Kafka consumer later
   Read historical events instantly.

7. Scale Kafka consumers
   Observe partition rebalance.

---

## BIGGEST LESSONS THIS PROJECT TEACHES

RabbitMQ answers:
"How do I reliably execute work?"

Kafka answers:
"How do I continuously stream business events?"

RabbitMQ is operational.

Kafka is informational.

RabbitMQ moves tasks.

Kafka stores business history in motion.

---

## FINAL MINDSET

RabbitMQ:
Task-oriented architecture.

Kafka:
Event-oriented architecture.

Large systems usually need BOTH.
