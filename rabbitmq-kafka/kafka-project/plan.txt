# Kafka Learning Project

Goal:

* Understand Kafka deeply by building a real event-driven system
* Compare Kafka mindset vs RabbitMQ mindset
* Learn partitions, offsets, replay, consumer groups, retention, scaling, retries, DLQ, ordering
* Build observable frontend + backend system to SEE how Kafka behaves

---

## PROJECT IDEA

Project name:
Realtime Commerce Event Platform

Business story:
Users place orders on an ecommerce system.

Instead of directly calling services synchronously:

* every action becomes an EVENT
* events are streamed through Kafka
* multiple services react independently

This project is intentionally designed to expose:

* event replay
* offset tracking
* consumer groups
* partitioning
* ordering
* retention
* retries
* DLQ
* scaling consumers
* stream processing mindset

## PHASE 1 — BASIC EVENT FLOW

Goal:
Understand producer/consumer basics.

Features:

1. Create Order API
2. Publish OrderCreated event
3. Multiple services consume same event

Flow:
Frontend
-> Order Service
-> publish "order.created"

Consumers:

* Payment Service
* Inventory Service
* Analytics Service
* Notification Service

What to learn:

* Kafka topic
* Producer
* Consumer
* Consumer group
* Event schema
* Event-driven architecture

Kafka concepts:

* one event can be consumed by MANY services
* event remains in Kafka
* services are decoupled

RabbitMQ comparison:
RabbitMQ usually thinks:
"deliver task"

Kafka thinks:
"publish event"

orders

3

* observe rebalance

What to learn:

* only one consumer per partition in same group

IMPORTANT:
Open Kafka UI and WATCH partition movement.

RabbitMQ comparison:

## PHASE 3 — OFFSETS

Goal:
Understand Kafka core concept.

Implement:
Frontend page:

* show current offsets
* show lag
* show processed events

Experiments:

1. Stop consumer
2. Produce many events
3. Restart consumer

Observe:

* consumer resumes from committed offset

Next:

* reset offset to beginning
* replay ALL events

What to learn:

* committed offset
* replay
* lag
* pull model

MOST IMPORTANT LESSON:
Kafka does NOT delete after consume.

## PHASE 5 — PARTITION + ORDERING

Goal:
Understand ordering guarantees.

Topic:
orders

Partition strategy:
key = user_id

Experiments:

* same user events stay ordered
* different users go to different partitions

Visualize:
Frontend page:

* partition distribution
* event timeline

What to learn:

* ordering only guaranteed INSIDE partition
* partition key importance

Critical lesson:
Bad partition key = bad system design.

## PHASE 6 — RETRY + DLQ

Goal:
Understand production-grade error handling.

Create:

* orders.retry.5s
* orders.retry.30s
* orders.dlq

Simulate:
Payment failures randomly.

Flow:
consume
-> fail
-> retry topic
-> fail many times
-> DLQ

Frontend:
DLQ dashboard

What to learn:

* Kafka retry pattern
* exponential retry
* dead letter queue
* poison messages

RabbitMQ comparison:
RabbitMQ has stronger built-in retry tooling.
Kafka retry usually requires explicit design.
